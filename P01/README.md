# 📁 P01: GitHub – Treballant de forma col·laborativa: Forks i Pull Request

## 📖 Introducció

Benvinguts a l’etapa de **treball col·laboratiu professional amb GitHub**.
Ara que cada membre de l’equip domina les eines de manera individual (servidors, permisos, configuració), toca **aprendre a treballar com un equip coordinat**.

A EverPia, el flux de treball segueix **bones pràctiques de Git/GitHub**:

* Jerarquia de rols (Caps d’Equip i Júniors)
* Branca principal `main` protegida
* Revisió obligatòria de codi abans de fusionar canvis (Pull Requests)
* Prohibició d’enviar arxius per correu

L’objectiu és que tots els membres aprenguin a:

* Fer forks del repositori central
* Crear branques de treball
* Realitzar commits amb missatges clars
* Obrir Pull Requests per a la revisió i aprovació del líder d’equip
* Gestionar comentaris i revisions de codi

---

## 🎯 Objectius de la tasca

1. Dominar el **flux col·laboratiu de GitHub**.
2. Treballar de manera coordinada dins d’un equip.
3. Simular un procés real d’**Onboarding** a EverPia.
4. Aplicar pràctiques professionals de control de versions i revisió de codi.

## 🔁 Flux de treball de GitHub a EverPia

1. **Fork del repositori base**

   * Cada membre fa un fork a la seva pròpia àrea de treball.
   * Això crea una còpia personal per treballar sense afectar el projecte central.

2. **Creació d’una branca de treball**

   * Sempre es treballa fora de `main`.
   * Exemple: `feat-config-server`, `update-docs`, `bugfix-login`.

3. **Realització de commits**

   * Fer commits petits i freqüents.
   * Missatges clars i descriptius:

     * `Afegit README inicial per onboarding`
     * `Configuració del servidor web completada`
     * `Correcció errors sintaxi en scripts`

4. **Sincronització amb el repositori base**

   * Actualitzar el fork amb els canvis de `main` del repositori central abans de crear un Pull Request.

5. **Pull Request (PR)**

   * Obrir un PR per fusionar els canvis a `main` del repositori central.
   * Assignar el "Team Leader" com a revisor.
   * Incloure una descripció clara del que s’ha fet i per què.

6. **Revisió de codi**

   * El líder d’equip revisa el PR:

     * Aprova si tot és correcte
     * Fa comentaris per millores si cal

7. **Merge a `main`**

   * Un cop aprovat, el PR es fusiona.
   * Manté `main` estable i funcional.

---

## 🧩 Rols dins l’equip

* **Team Leader (Líder d’Equip)**

  * Revisió de Pull Requests
  * Aprovar o demanar millores
  * Coordinació del flux de treball

* **Membres de l’equip**

  * Treball en branques pròpies
  * Creació de commits i PR
  * Seguiment de les revisions del líder

---

## 📌 Bones pràctiques

* Evitar commits grans amb canvis massius.
* Descriure clarament cada commit.
* Sincronitzar el fork abans de crear un PR.
* Fer revisions constructives i documentar els canvis.

---

## ✅ Producte final

* Cada membre ha de tenir **un fork actualitzat**.
* Haver creat **almenys un PR revisat i fusionat** pel líder d’equip.
* Documentar el procés en un **README.md** dins la carpeta corresponent del repositori del projecte.
* Mostrar evidències: captures de pantalla dels PR aprovats i merges realitzats.

---

## 🏁 Conclusió

Dominar forks i Pull Requests és essencial per treballar de manera **col·laborativa i professional** en qualsevol empresa de desenvolupament de software.
Aquest procés garanteix:

* Control de qualitat del codi
* Coordinació entre membres
* Manteniment d’un repositori estable
* Preparació per al món laboral real
